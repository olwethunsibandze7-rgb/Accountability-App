# Table: `profiles`

## Purpose

Stores the persistent app-level identity and onboarding/account settings for each authenticated user.

## SQL structure

```text
id uuid PK
timezone text NOT NULL DEFAULT 'UTC'
created_at timestamptz DEFAULT now()
setup_completed boolean DEFAULT false
username text
updated_at timestamptz DEFAULT now()
plan_tier text NOT NULL DEFAULT 'free'
strict_mode_enabled boolean NOT NULL DEFAULT true
wake_time time NOT NULL DEFAULT 06:00
sleep_time time NOT NULL DEFAULT 22:00
onboarding_step smallint NOT NULL DEFAULT 0
```

## Constraints

* `profiles_pkey` â†’ primary key on `id`
* `profiles_id_fkey` â†’ `id references auth.users(id) on delete cascade`
* `profiles_onboarding_step_check` â†’ onboarding step must be `0..5`
* `profiles_plan_tier_check` â†’ must be `free` or `pro`
* `profiles_sleep_after_wake_check` â†’ `sleep_time > wake_time`

## Trigger

* `set_profiles_updated_at`
* fires `before update`
* executes `set_updated_at()`

## Meaning in product

This is the userâ€™s main Achievr identity record and the bridge between Supabase Auth and app data.

# Table: `habit_schedules`

## Purpose

Stores recurring weekly schedule templates for each habit.

## SQL structure

## Constraints

* `habit_schedules_pkey` â†’ primary key on `schedule_id`

* `habit_schedules_unique_slot` â†’ unique on `(habit_id, day_of_week, start_time, end_time)`
* `habit_schedules_check` â†’ `end_time > start_time`
* `habit_schedules_day_of_week_check` â†’ day must be `1..7`

## Meaning in product

These rows are created from onboarding scheduling logic and used to generate daily logs.

## Constraints

* `habit_logs_habit_id_fkey` â†’ `habit_id references habits(habit_id) on delete cascade`
* `habit_logs_schedule_id_fkey` â†’ `schedule_id references habit_schedules(schedule_id) on delete set null`
* `habit_logs_user_id_fkey` â†’ `user_id references auth.users(id) on delete cascade`

## Meaning in product

Each row represents one actual daily habit occurrence.

* Today screen execution

## Notes

That is redundant, but not invalid.

# Current Cascade Delete Behavior

## Root delete

When a row is deleted from:

```text
auth.users
```

this cascades to:

```text
profiles
habit_logs
```

because:

* `profiles.id -> auth.users(id) on delete cascade`
* `habit_logs.user_id -> auth.users(id) on delete cascade`

Then `profiles` cascades into:

```text
goals
fixed_time_blocks
verification_records
```

Then `goals` cascades into:

```text
habits
```

Then `habits` cascades into:

```text
habit_schedules
habit_logs
verification_records
```

## Practical result

Deleting a Supabase auth user should remove all of that userâ€™s related Achievr data:

* profile
* goals
* habits
* habit schedules
* habit logs
* fixed time blocks
* verification records

So this is now valid for onboarding reset testing.

# Current Schema Notes / Engineering Observations

## 1. `habit_logs.user_id` is somewhat redundant

Because `habit_logs` already references `habits`, and habits reference goals, and goals tie back to user/profile, the direct `user_id -> auth.users` link is not strictly necessary for relational cleanup.

It may still be useful for:

* simpler queries
* faster filtering by user
* easier security rules

So it is acceptable.

## 2. `verification_records.user_id` points to `profiles(id)`

That is consistent with your profile-centered app data structure.

## 3. `fixed_time_blocks.user_id` and `goals.user_id` point to `profiles(id)`

That is also consistent and clean.

## 4. `habit_logs.schedule_id` uses `on delete set null`

This is a reasonable choice because a historical log can conceptually survive even if the template schedule row changes.
In full user deletion, the logs are still removed through other cascade paths.
