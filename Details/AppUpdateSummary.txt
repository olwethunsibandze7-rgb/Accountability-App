# Achievr App — Project State (March 2026)

## Project Overview

**Achievr** is a goal-enforcement productivity app built to create **accountability through scheduled habits tied directly to user goals**.

Instead of generic to-do lists, the system works like this:

1. Users define **2 core goals**
2. Each goal generates **3 enforced habits**
3. Habits are scheduled automatically around **blocked time**
4. Daily habit instances are generated into **habit logs**
5. Users verify completion against scheduled execution windows
6. Missed or incomplete habits will later be recorded to enforce accountability

The philosophy is:

> Goals are meaningless unless enforced through structured daily actions.

# SQL / Backend Logic Added Since Last Snapshot

## Daily log generation function

A Postgres RPC function now exists:

```text
generate_daily_habit_logs(p_user_id uuid, p_date date)
```

It:

* determines the weekday from the provided date
* reads that user’s active goals and habits
* finds matching `habit_schedules`
* inserts daily rows into `habit_logs`
* includes:

  * `user_id`
  * `habit_id`
  * `schedule_id`
  * `log_date`
  * `scheduled_start`
  * `scheduled_end`
  * `status`
  * `verification_type`

This is the foundation of the real accountability engine.

### Verification rule currently implemented

For `manual` verification:

* task can only be completed during scheduled execution window
* plus a **30-minute grace period** after scheduled end

* user cannot mark a 4–6 PM task done at 12 AM

# Important Product Rules Currently Implemented

## Manual verification lock

For manual tasks:

* completion is only allowed inside the scheduled window
* plus 30-minute grace period

This prevents cheating and keeps accountability meaningful.

## Refresh behavior

Refresh was intentionally made less sensitive:

* top-edge pull only
* larger displacement
* less accidental reload feel

# Current Known Engineering Notes

## 1. Time logic still uses real clock

The app currently uses real `DateTime.now()` in multiple places.

A future improvement is to create an `AppClock` / debug clock service so time-dependent logic can be tested without changing emulator system time.

This would affect:

* Today verification windows
* next task logic
* progress date logic
* Upcoming ordering
* habit log generation date handling

## 2. Streak logic is MVP-level

Current streak logic works, but is not yet strict enough for final production logic.

## 3. Verification system is designed for extension

Current implementation supports `manual`, and UI/service structure is intentionally being kept extensible for:

* `location`
* `third_party`
* other proof systems later
